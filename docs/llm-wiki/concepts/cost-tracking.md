# Cost tracking and session model

**Purpose** Account for per-turn spend, surface it as a running session/daily total in the toolbar, and capture the real session model for display. Cost is the CLI's reported `total_cost_usd` when present, else a **notional cost computed from token usage** via a local price table (subscription/Bedrock plans report `0`). Both the cost figure and the captured model are **accounting/display only** — neither ever feeds back into how the next CLI process is launched.

## Invariants

- **The captured session model is never used to select the model.** `EventStreamHandler.currentModel` is *seeded* from `CliEvent.SystemInit.model`, then *refreshed* from each top-level `CliEvent.AssistantMessage.model` (parsed by `CliEventParser` from the nested `message.model`). The assistant-message value is the authoritative source for the footer because `SystemInit.model` is blank on resume and absent in some CC init shapes; `currentModel` is used only for the per-turn footer and the `perModelUsd` accounting key. The model actually passed to the CLI is resolved independently in `CliProcess` from `ClawDEASettings.getCliModelId(...)` → `buildModelArg` → `--model`. There is no path from `currentModel` (or the resumed transcript) to `--model` ([EventStreamHandler.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/EventStreamHandler.kt), [CliProcess.kt](../../../src/main/kotlin/com/adobe/clawdea/cli/CliProcess.kt)).
- **Per-turn cost is summed, not overwritten — and falls back to a token-priced figure.** `recordTurn` computes the *effective* turn cost via `effectiveTurnCost`: the CLI's `total_cost_usd` (`CliEvent.Result.costUsd`, parsed from the end-of-turn field) when it is `> 0`, otherwise `ModelPricing.costFor(model, …)` over the turn's `usage` token breakdown. The effective figure is *summed* into `sessionUsd`, `perModelUsd`, and the daily total. (Contrast with `contextTokens`, which `EventStreamHandler` overwrites because it is already a per-turn context-size figure — see [CLI bridge](cli-bridge.md).) ([CliEventParser.kt](../../../src/main/kotlin/com/adobe/clawdea/cli/CliEventParser.kt), [CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [ModelPricing.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/ModelPricing.kt)).
- **The token-priced fallback exists because subscription/Bedrock plans report `total_cost_usd = 0`.** `CliEvent.Result` carries the per-turn `inputTokens`/`outputTokens`/`cacheReadTokens`/`cacheCreationTokens` from the result `usage` object precisely so the chip keeps moving on flat-rate plans. `ModelPricing` holds published per-million-token rates (longest-prefix model match — plus a date-aware branch matched *ahead* of the prefix list for Sonnet 5, whose introductory rate reverts to the standard Sonnet tier on 2026-09-01 — with a conservative non-zero fallback for unknown ids; cache-read billed at 0.1× input, cache-creation at 1.25×). `recordTurn`'s `contextTokens` parameter remains intentionally unused (`@Suppress("UNUSED_PARAMETER")`) — context size is not a cost input ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [ModelPricing.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/ModelPricing.kt)).
- **Session total is per-chat-tab; the daily total is application-wide.** `CostTracker` is a `@Service(PROJECT)`, but it no longer holds a single `sessionUsd` — it keeps a `chats: Map<String, ChatCost>` keyed by a stable per-tab `chatId`, each `ChatCost` carrying its own `sessionUsd` and `perModelUsd`. Every entry point (`recordTurn`, `seedSession`, `resetSession`, `seedFromResume`, `snapshot`) takes a `chatId` and operates on `chat(chatId)` (lazily created via `getOrPut`). `ChatPanel` mints the id once at construction — `"chat-" + System.identityHashCode(this).toString(16)` — because the CLI session id changes on resume, so cost must be keyed by panel lifetime instead. Opening a second chat tab gives it its own session counter. The daily total is broader: it lives on the **application-level** `ClawDEASettings` and accumulates across all chats and projects ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [ChatPanel.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/ChatPanel.kt)).
- **"Default (<model>)" is labeled from an observed turn, never from a selection or from history.** `CostTracker` holds a project-wide `defaultResolvedModel`, set in `recordTurn` *only* when the turn ran with `ranUnderDefaultSelection = true` (no explicit `--model`, so the CLI chose). It is deliberately **not** set from explicit selections or from resumed transcripts (whose model reflects a past choice, not today's default), and `resetSession` does **not** clear it — what Default resolves to is a provider fact, not session state. `ChatPanel` computes `ranUnderDefaultSelection` from the same condition under which `buildModelArg` omits `--model` (`getSelectedModelId(...).isBlank()`). The resolved model rides along in `CostSnapshot.defaultResolvedModel`; `ModelComboManager` reads `CostTracker.defaultResolvedModel()` to label its Default entry as `Default (Opus 4.8)` (via `defaultLabel`/`prettyModelName`) and refreshes the combo only when the resolved value actually changes ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [ModelComboManager.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/ModelComboManager.kt)).
- **The daily total rolls over by date, not by clock.** `rolledDaily` resets the accumulator whenever `dailyCostDate != today`; there is no timer. A turn on a new day is the trigger. The read-modify-write on the shared `ClawDEASettings` is wrapped in `synchronized(settings)` so concurrent projects don't lose updates ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt)).
- **Resume reseeds the chat's session from the transcript and returns the figure for a footer.** `seedFromResume(chatId, sessionId)` calls `resetSession(chatId)`, then reconstructs the prior conversation's cost from its `.jsonl` via `TranscriptCostReader.readResumeCost`, seeds that chat's total as a baseline (`seedSession(chatId, …)`), and **returns a `TranscriptCostReader.ResumeCost(totalUsd, lastModel)`**. It deliberately does **not** set `defaultResolvedModel` from the transcript. `SessionManager.resumeSession` uses that return value: when `totalUsd > 0` it renders a per-turn-style cost footer (`renderer.renderCostInfo(lastModel, effort = null, totalUsd, elapsed = 0)`) so a reopened conversation shows its accumulated cost at the bottom of the chat, not just on the toolbar chip. This restores the running session figure after a resume; it does **not** touch the daily total (which already accumulated when those turns originally ran) ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [SessionManager.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/SessionManager.kt)).
- **Transcript reading prices token usage and de-duplicates streamed copies by message id.** `TranscriptCostReader.readResumeCost` (which `sumCost` now delegates to) calls `priceLine(line, lineIndex, seen)` on every line: lines without a `"usage"` object (or without a `"model"`) return `null` and contribute `0`; the rest are priced through `ModelPricing.costFor` over their `input`/`output`/`cache_read`/`cache_creation` token counts. Crucially, Claude Code rewrites each assistant message to the `.jsonl` multiple times while streaming (observed up to ~4 byte-identical copies sharing one `message.id`), so `priceLine` keys each line on its `message.id` (falling back to a per-line key when absent) and a **repeat id is priced once** — a duplicate returns `0.0` cost but still reports its model, so the footer reflects the latest turn. Without this de-dup, resume cost over-counted by ~3×. This mirrors the live `effectiveTurnCost` fallback so a resumed session reconstructs the same notional total. Extraction is hand-rolled string scanning (no JSON parse); any exception swallows to `0.0` because seeding is non-critical. The transcript path scheme (`~/.claude/projects/-<encoded-base-path>/<sessionId>.jsonl`) mirrors `SessionScanner` and must stay in sync with it ([TranscriptCostReader.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/TranscriptCostReader.kt), [ModelPricing.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/ModelPricing.kt)).
- **The warning band is provider-dependent and, for subscription, driven by live usage — not a separate window struct.** `snapshot` bands off `bandForUsage(usage)` **only** when `providerId == "subscription" && usage.available`; otherwise it falls back to `bandForDollars` (daily spend vs the user's daily budget). `bandForUsage(u: SubscriptionUsage)` takes the worst utilization across the spend gauge (`u.spend?.pct`) and every window (`u.windows[].pct`): `≥ 90` → `RED`, `≥ 75` → `AMBER`, else `GREEN`, and `NEUTRAL` when no percentages are present. `bandForDollars` returns `NEUTRAL` for a daily budget of `0` ("no budget"). The old `SubscriptionWindow` type, the `CostTracker.window` field + `updateWindow`, `CostSnapshot.window`, and `bandForWindow` were **removed** — after the `oauth/usage` work `updateWindow` was never called, so the window branches were dead and the subscription band silently fell back to the budget-only path (gray with no budget). Live `SubscriptionUsage` (fed by `updateUsage`) is now the single source for both the band and the chip text ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt)).

- **Provider cumulative totals, knowledge-bucket sub-totals, and live `oauth/usage` are all wired into the cost path.** Three additional `CostSnapshot` fields back the cost-visibility work: `providerTotal` (per-provider `sinceDate`/`monthToDate`/`allTime`, persisted packed as `"since|mtd|allTime|mtdMonth"` in `ClawDEASettings.state.providerTotals` and rolled by calendar month via `ProviderTotal.add`), `knowledgeUsd` (per-`KnowledgeBucket` spend for wiki/workspace upkeep turns), and `usage` (a `SubscriptionUsage` parsed from the Anthropic `oauth/usage` endpoint by `OAuthUsageClient`). **Provider/knowledge attribution:** `EventStreamHandler`'s `recordTurn` call passes `providerId = AuthManager.effectiveProviderId()` and `knowledgeBucket`, so `recordTurn` folds the effective turn cost into `addToProvider(providerId, …)` and (when non-null) `addToKnowledge(bucket, …)` on every turn. **`knowledgeUsd` is GLOBAL (app-level), not per-chat or per-project** — wiki/workspace upkeep comes from chat-dispatched commands *and* a background drift subprocess that belongs to no chat, *and* from any project, so it is accrued into the application-level `ClawDEASettings.state.knowledgeUsd` (keyed by `KnowledgeBucket.name` → cumulative USD) via `addToKnowledge`, and read back through `readKnowledge` (which drops unknown bucket keys). It reads as one upkeep bill across every project and shows in every tab's Cost Control panel (a breakdown of overall spend, not a separate pool). **Bucket classification has two sources.** A bucket is parked in `EventStreamHandler.pendingKnowledgeBucket` at submit time and consumed (and cleared) when the turn's `Result` arrives — submit-time capture is required because the result event carries no prompt text — but the value can arrive two ways: (1) for raw user turns, `onTurnSubmitted(text)` falls back to `KnowledgeBucketClassifier.classify(promptText)`; (2) for **slash commands**, the bucket is classified by command *name*, not prompt text, because a command like `/seed-wiki` dispatches a long *expanded template* (not the literal command) to the bridge, which the text classifier would always miss. `ChatPanel` computes `KnowledgeBucketClassifier.classifyCommand(handler.info.name)` (`info.name` is the slash form, e.g. `"/seed-wiki"`; `classifyCommand` delegates to `classify`) at the routing site and threads that explicit bucket through `buildCommandContext(bucket)` → `dispatchSendToBridge(text, …, knowledgeBucket = bucket)` → `onTurnSubmitted(text, explicitBucket)`, where `explicitBucket` overrides the text fallback. The drift-review dispatch hard-tags `KnowledgeBucket.WIKI_UPDATE` the same way. **Live usage polling is a single application-wide poll, fanned out to every project.** `OAuthUsageService` is an **application-level `@Service(Service.Level.APP)`** — exactly **one** poll loop runs for the whole IDE (not one per project, and not one per chat tab), guarded by an idempotent `start()` (`AtomicBoolean`). It runs on a pooled thread and calls `OAuthUsageClient.fetch()` → `OAuthUsageCache.publish(usage)` on a ~5-minute cadence, **not gated on the active provider** — `fetch()` self-limits, returning `UNAVAILABLE` when no OAuth token is present, so a Bedrock (or API-key) user who *also* has a Claude subscription still sees its usage in the panel. It backs off once on failure — `5m → 10m`, held at a 10m ceiling (`OAuthUsageService.backoffMs`, `FAILURE_CAP = 1`) — and publishes `UNAVAILABLE` so the UI falls back to the notional estimate. `OAuthUsageClient.fetch()` logs the granular failure cause (HTTP status / exception class, never the token) and the service logs availability transitions, so a flip to unavailable is diagnosable instead of silent. **The single shared poll exists to stop per-project divergence:** the former per-project poller ran N independent loops over one rate-limited endpoint, so a transient failure (timeout, momentary non-200, token-read race) flipped only that project to `UNAVAILABLE` and backed it off for up to 40 minutes while sibling projects whose polls happened to succeed kept showing usage. `OAuthUsageCache` holds the latest value and fans it out: `subscribe(sink)` **replays the latest cached value immediately**, so a project opening (or recovering) mid-cycle shows usage at once instead of waiting for the next fetch. `OAuthUsageProjectLink` is the **per-project `@Service`** bridge: on `connect()` it subscribes this project's `CostTracker.updateUsage(usage)` to the shared cache and starts the (idempotent) shared poll; the subscription is torn down when the project (this link) disposes. `OAuthUsagePollerStartupActivity` (a `ProjectActivity`) calls `OAuthUsageProjectLink.connect()` on project open. So `usage` is genuinely populated whenever an OAuth/subscription token exists; it stays `UNAVAILABLE` when there is no subscription token and until the first successful poll ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [ProviderTotal.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/ProviderTotal.kt), [OAuthUsageClient.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/OAuthUsageClient.kt), [OAuthUsageService.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/OAuthUsageService.kt), [OAuthUsageCache.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/OAuthUsageCache.kt), [OAuthUsageProjectLink.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/OAuthUsageProjectLink.kt), [KnowledgeBucket.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/KnowledgeBucket.kt), [ChatPanel.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/ChatPanel.kt), [EventStreamHandler.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/EventStreamHandler.kt)).

- **The `oauth/usage` shape is a spend gauge *and* a window list, not one-or-the-other.** The endpoint returns a flat object of rate-limit window keys (each `null` or `{utilization, resets_at}`) plus an `extra_usage` spend object — both may be present at once. `OAuthUsageClient.parse` surfaces `SubscriptionUsage.Spend(used, limit, pct, currency)` from `extra_usage.{used_credits, monthly_limit, utilization, currency}` as the headline gauge, and a `List<UsageWindow>` built **only** from an allowlist (`WINDOW_LABELS`: `five_hour`, `seven_day`, `seven_day_opus`, `seven_day_sonnet`) — the endpoint's internal codename keys (`tangelo`, `iguana_necktie`, `cinder_cove`, `*_omelette`, …) are deliberately dropped so the UI never shows them. `utilization` is a Double percent (rounded via `Math.round`); **`used_credits`/`monthly_limit` are in *cents* of `currency`** (e.g. `54652` = `$546.52`, `60000` = `$600.00`), so `parse` **divides each by 100** — `Spend.used`/`limit` are therefore canonical dollar amounts everywhere downstream (`CostControlPanel` renders them with a `$` prefix via `money2` *plus* the `currency` code, e.g. `$546.52 of $600.00 USD`). `parse` returns `UNAVAILABLE` only when *neither* a spend gauge *nor* any known window is present. The schema is undocumented and drift-watched against `src/test/resources/oauth-usage/live-sample.json` ([CostModels.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostModels.kt), [OAuthUsageClient.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/OAuthUsageClient.kt)).

- **The multi-provider Cost Control panel is fed by one `ProviderBlock` per *used* provider, not just the active one — and subscription appears whenever live usage is available.** `CostTracker.providerBlocks()` returns a `List<ProviderBlock(providerId, total: ProviderTotal?, usage: SubscriptionUsage)>` — its header rows. The set of providers is built in two steps: (1) the pure companion helper `usedProviders(stored: Set<String>, active: String)` unions the **persisted** `ClawDEASettings.state.providerTotals` keys with the **currently active** provider (`AuthManager.effectiveProviderId()`), preserving insertion order and dropping a blank active id; (2) `providerBlocks()` then **adds `"subscription"` whenever `usage.available`**, even if the user is not currently on subscription and has no persisted subscription total. So a provider the user has spent on before still gets a block even when it isn't the one in use right now, the active provider appears even before its first persisted total, *and* a Bedrock (or API-key) user who also has a Claude subscription still sees the subscription spend gauge — which is the whole point of polling `oauth/usage` regardless of the active provider. The live `usage` gauge attaches **only** to the `"subscription"` block (others get `SubscriptionUsage.UNAVAILABLE`); each block's `total` is the parsed per-provider `ProviderTotal`, and the chat-scoped body (session/per-model/knowledge) still comes from `snapshot(chatId)` — `providerBlocks()` is header data only. Splitting provider enumeration into the testable pure `usedProviders` keeps the union rule independent of the project service; the `usage.available` add-on is layered on top in the service method ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [CostModels.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostModels.kt), [ProviderTotal.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/ProviderTotal.kt)).

- **Cost is recorded only at turn granularity — there is no per-tool or per-subagent attribution today.** The single accounting trigger is `CliEvent.Result` (the end-of-turn `result` event), which carries `costUsd` and the turn's aggregate token usage but **no breakdown by tool call or sub-agent**. `recordTurn` therefore attributes a turn's cost only along the dimensions it is given — `chatId`, `model`, `providerId`, and `knowledgeBucket` — never to the individual tools or sub-agents that ran inside the turn. The event stream *does* carry the data a per-tool/per-subagent estimator would need: `CliEvent.AssistantMessage.toolUses` (each `ToolUse` id/name/input), `CliEvent.ToolResult`, and the `parentToolUseId` field threaded through `AssistantMessage`/`ToolResult`/`TextDelta` — which `SubAgentController` already consumes to group a sub-agent's inner events under one card. But that attribution lives entirely on the rendering/grouping side; it is **not** wired into the cost path, and `Result` (the only event `recordTurn` sees) does not include it. Anyone designing per-tool or per-subagent cost estimation must add a new accumulation path keyed off the stream events — the present cost figures cannot be decomposed below the turn ([CliEvent.kt](../../../src/main/kotlin/com/adobe/clawdea/cli/CliEvent.kt), [EventStreamHandler.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/EventStreamHandler.kt), [SubAgentController.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/SubAgentController.kt)).

- **The background drift wiki-author subprocess is metered separately from any chat.** The auto-applied [drift](drift-detection.md) digest runs as its own `claude -p` subprocess (`DefaultWikiAuthorInvoker`), tied to no chat, so its cost would otherwise be discarded. `DriftDetectionService` constructs the invoker with an `onStdout` callback that routes the subprocess's stream-json output to `CostTracker.recordWikiAuthorCost(stdout)`; `DefaultWikiAuthorInvoker` invokes `onStdout(result.stdout)` after every run when stdout is non-blank — **regardless of exit code** (a timed-out or non-zero run may still have produced billable turns), guarded by try/catch so a metering failure never breaks drift. `recordWikiAuthorCost` calls the companion `parseResultLine(stdout)`, which scans the lines for the trailing `result` event (cost + token usage) and back-fills the model id from the last `AssistantMessage.model` (the result event carries no model). It computes `effectiveTurnCost` and, when `> 0`, records to the **`WIKI_UPDATE`** knowledge bucket (via `addToKnowledge`, which guards the global `ClawDEASettings.state.knowledgeUsd` under `synchronized(settings)`), the active provider's total, and the daily total — but deliberately **not** to any chat's `sessionUsd`, since the subprocess belongs to no tab. Both `recordWikiAuthorCost` and the test-covered `parseResultLine`/`ParsedResult` live in `CostTracker` ([CostTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostTracker.kt), [WikiAuthorInvoker.kt](../../../src/main/kotlin/com/adobe/clawdea/knowledge/drift/WikiAuthorInvoker.kt), [DriftDetectionService.kt](../../../src/main/kotlin/com/adobe/clawdea/knowledge/drift/DriftDetectionService.kt)).

## Resolution pipeline

1. The CLI emits a `result` event at end of turn. `CliEventParser` extracts `total_cost_usd` into `CliEvent.Result.costUsd` and the `usage` token breakdown into `inputTokens`/`outputTokens`/`cacheReadTokens`/`cacheCreationTokens`.
2. At submit time (before each `bridge.sendMessage`), `ChatPanel` calls `EventStreamHandler.onTurnSubmitted(text, explicitBucket?)`, which parks a knowledge bucket in `pendingKnowledgeBucket` for this turn: the `explicitBucket` when one was threaded through (slash commands classify by command name — `classifyCommand(handler.info.name)` — and pass it via `buildCommandContext`/`dispatchSendToBridge`), otherwise a `KnowledgeBucketClassifier.classify(text)` fallback for raw user turns.
3. `EventStreamHandler` (on `CliEvent.Result`) renders the per-turn footer (`renderCostInfo(model, effort, costUsd, elapsed)`) using `currentModel` for the model label, then calls `CostTracker.recordTurn(chatId, currentModel, event.costUsd, event.contextTokens, …, ranUnderDefaultSelection = resolveIsDefaultModel(), providerId = AuthManager.effectiveProviderId(), knowledgeBucket = pendingKnowledgeBucket)` and clears `pendingKnowledgeBucket`. The handler holds the panel's `chatId` and a `resolveIsDefaultModel` callback (true ⇔ no explicit model stored for the provider/dir).
4. `recordTurn` computes the effective cost (`effectiveTurnCost`: reported `total_cost_usd` if `> 0`, else `ModelPricing.costFor` over the tokens) and, when `> 0`, adds it to that chat's `sessionUsd`, merges into its `perModelUsd` keyed by model, folds into the per-provider total (`addToProvider`) and the **global, app-level** knowledge bucket map (`addToKnowledge(bucket, …)` into `ClawDEASettings.state.knowledgeUsd`, when a bucket was classified), and folds into the application-level daily total. When `ranUnderDefaultSelection` is true it also records `defaultResolvedModel = model`. Then it `publish()`es. (The background drift subprocess takes a parallel route: `recordWikiAuthorCost` folds its cost into the same global `WIKI_UPDATE` bucket + provider + daily totals, but no `sessionUsd`.)
5. `publish()` is a **payload-free signal**: it fires `CostSnapshotListener.TOPIC.onCostChanged()` on the project message bus **synchronously while the instance lock is held** — listeners must be quick and must not call back into `CostTracker`. The snapshot is per-chat now, so the payload can't be a single object; each listener re-queries `snapshot(itsChatId)` instead. `CostTracker.refresh()` is a thin public alias for `publish()` — a re-broadcast with no turn behind it — for settings changes that bypass `recordTurn`, notably the daily-budget edit in step 6's footer.
6. `CostChip` (toolbar `JBLabel`, constructed with its panel's `chatId`) handles `onCostChanged` by calling `CostTracker.snapshot(chatId)`, then hops to the EDT via `invokeLater` and re-renders label text (provider-aware via `formatText`), color (`bandColor`), and tooltip. Clicking the chip opens the **Cost Control panel** (`CostControlPanel(project, chatId).showUnder(chip)`) — a focused, **movable and resizable** popup (`setMovable(true)`/`setResizable(true)`) that **centers over the ClawDEA tool window** (`showInCenterOf(host)`, falling back to `showUnderneathOf(anchor)` only when the tool window isn't showing) rather than dropping below the chip, where it was opening clipped off the bottom of the view. Its content is wrapped in a `JBScrollPane` and auto-sized to its preferred height (capped at the tool-window height minus a margin) so a tall breakdown is never clipped and needs no manual resize. The layout is **card-based** (one rounded `RoundedPanel` per section): one **provider card** per used provider (`providerBlocks()`; subscription gets a colored spend gauge, others get month-to-date/all-time + a per-provider Reset button), a **"This chat" card** (`snapshot(chatId)`: session `Cost` + a By-model breakdown), a standalone **"Knowledge upkeep" card**, and a **daily-budget footer card**. Each card always renders completely: the Knowledge-upkeep card iterates *all four* `KnowledgeBucket.entries` and prints `$0.00` for any bucket with no spend (rather than skipping absent ones), and the By-model section emits a `(no per-model data yet)` placeholder when `perModelUsd` is empty instead of vanishing. Setting the budget moved *into* that footer; the chip click no longer prompts for it directly. The footer's "Set daily budget" button writes `ClawDEASettings.state.dailyBudgetUsd` and then calls `CostTracker.refresh()`, so every open chip re-snapshots and re-bands against the new budget immediately rather than waiting for a new chat to rebuild it (the daily budget feeds `bandForDollars`, which is what the chip color reflects on non-subscription providers). `ModelComboManager` is the other listener — it re-labels its Default entry when `defaultResolvedModel()` changes.
7. On resume, `SessionManager.resumeSession` calls `CostTracker.seedFromResume(chatId, sessionId)` *before* restarting the bridge, so the chip shows the restored session total for that tab immediately. The returned `ResumeCost` (de-duplicated total + last model) also drives a one-off `renderCostInfo` footer appended to the chat history (`effort = null`, `elapsed = 0`), mirroring the live per-turn footer.

## Anti-patterns

- **Feeding the captured/display model back into `--model`** — `currentModel` and the resumed transcript are accounting/display inputs only. The launch model is resolved from settings; do not wire `SystemInit.model` into command assembly or a resume would silently pin the model.
- **Treating `total_cost_usd` as cumulative, or token-pricing when a real dollar figure exists** — `total_cost_usd` is per-turn; `recordTurn` sums turns. `ModelPricing` is a *fallback* used only when the reported cost is `0` (flat-rate plans); never recompute from tokens when the CLI already reported a non-zero dollar figure, or you'll diverge from the authoritative number.
- **Reseeding the daily total on resume** — `seedFromResume` only reseeds the one chat's session view. The daily total already accumulated when those turns ran; reseeding it would double-count.
- **Labeling "Default" from an explicit selection or from resumed history** — `defaultResolvedModel` is set only by a turn that ran with `ranUnderDefaultSelection = true`. An explicit pick tells you nothing about what Default resolves to, and a transcript's model is a past choice; either would mislabel the selector.
- **Pricing every transcript line** — Claude Code writes each assistant message to the `.jsonl` several times while streaming (same `message.id`, identical `usage`). Summing every line over-counts resume cost by ~3×. De-duplicate by `message.id` (the `seen` set in `priceLine`) and price each logical turn once.
- **Heavy or re-entrant `CostSnapshotListener` work** — listeners run synchronously under the `CostTracker` lock. Block or call back in and you deadlock or stall the turn. Marshal to the EDT and return fast (as `CostChip` does).
- **Mutating cost-affecting settings without republishing** — chips/panels only re-snapshot on `onCostChanged`. A change that bypasses `recordTurn` (e.g. editing `dailyBudgetUsd` in the panel footer) leaves every open chip on its stale band/budget until a new chat rebuilds it. Call `CostTracker.refresh()` after the write so listeners re-band at once.
- **Treating the session total as per-project and shared across tabs** — it is now per-chat-tab, keyed by `chatId` in the `chats` map; each tab is isolated. Only the daily total is application-wide. A `recordTurn`/`snapshot`/`seedFromResume` call without the right `chatId` lands on (or reads) the wrong tab's counter.
