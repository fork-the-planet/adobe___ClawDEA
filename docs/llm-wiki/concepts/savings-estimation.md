# Savings estimation

**Purpose** Estimate what ClawDEA saves (or costs) relative to standard Claude Code, turn by turn, and surface a net "ClawDEA Estimated Savings / Cost" figure in the Cost Control panel. Every figure is anchored in **real observed tokens**; only the counterfactual — "what standard CC would have spent" — is modeled, and only those modeled pieces carry a confidence band. This is a parallel accounting path to [cost tracking](cost-tracking.md): it runs off the same turn events but answers a different question (delta vs. standard CC, not absolute spend).

## Invariants

- **Sign convention: positive = saving, negative = cost.** A `SavingsBand(low, expected, high)` is a signed estimate where `low <= expected <= high` always holds. Negation swaps edges (`unaryMinus`) so the invariant survives. Levers that are pure costs (primer overhead, knowledge upkeep) contribute negative bands ([SavingsModels.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsModels.kt), [SavingsEstimator.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsEstimator.kt)).
- **Four levers, fixed display order.** `LeverId` enumerates `LIBRARIAN`, `INDEX_TOOLS`, `KNOWLEDGE_UPKEEP`, `PRIMER_OVERHEAD` and its declaration order is the panel's display order. Two levers are **modeled** (band width > 0): `LIBRARIAN` (subagent routing) and `INDEX_TOOLS` (IDE index tools replacing grep+read). Two are **measured** (zero-variance, `measured = true`, `low == expected == high`): `PRIMER_OVERHEAD` and `KNOWLEDGE_UPKEEP` — real costs read straight from token/dollar accounting ([SavingsModels.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsModels.kt), [SavingsEstimator.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsEstimator.kt)).
- **`SavingsEstimator` is pure.** Every method is a pure function of a `TurnObservation`; no IntelliJ deps, fully headless-testable. `components(obs)` returns all four `SavingsComponent`s in display order; `aggregate(obs)` sums their bands into the turn net ([SavingsEstimator.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsEstimator.kt)).
- **The librarian lever is an honest counterfactual and CAN be negative.** For each observed subagent, the saving = *avoided re-rides* net of *the subagent's own real cost*. Had the main loop explored inline, it would have paid the first read at full input rate (same as the subagent did) and then RE-RIDDEN those tokens in the main context for the remaining turns at cache-read rate. So the band is: `low` = first read only (no re-rides materialise), `expected` = re-rides at cache-read rate × `remainingTurns`, `high` = re-rides at cache-creation rate. Minus `s.costUsd`. A one-shot question (`remainingTurns == 0`) avoids no re-rides, and an expensive subagent that read little nets negative — both honest outcomes, deliberately not floored at zero ([SavingsEstimator.librarian](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsEstimator.kt)).
- **`remainingTurns` drives the re-ride multiplier, and the live path cannot see the future.** The live path uses the **prior** turn count as a proxy floor (`priorTurns = savingsTracker.snapshot(chatId).turnCount`), because at the moment a turn's `Result` arrives the session's future length is unknown. The transcript reseed path, by contrast, has the whole session on disk and passes each turn's *real* `totalTurns - turnIndex - 1` — the accurate estimate ([EventStreamHandler.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/EventStreamHandler.kt), [TranscriptSavingsReader.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/TranscriptSavingsReader.kt)).
- **The index-tools lever is priced at cache-read rate.** IDE `find_*` / `resolve_symbol` / `get_diagnostics` results replace a grep + file reads that would have ridden the main context. Avoided = the token size of the hit files: `low` = one hit's worth, `expected` = all hits, `high` = all hits × `GREP_SCAN_FACTOR` (1.5, the avoided full-tree scan pass). `TurnObservationBuilder.isIndexTool` matches on the **bare** name (`substringAfterLast("__")`) so both bare and `mcp__clawdea-intellij__`-namespaced tool names count ([SavingsEstimator.indexTools](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsEstimator.kt), [TurnObservationBuilder.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/TurnObservationBuilder.kt)).
- **Primer overhead is a measured cost, not an estimate.** The extra primer tokens (wiki TOC + REPO_STATE) ClawDEA ships every turn that standard CC would not, priced exactly from the cache-read vs cache-creation split via the same `ModelPricing` multipliers. `SavingsBand.exact(-cost)`, `measured = true` ([SavingsEstimator.primerOverhead](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsEstimator.kt)).
- **Session band is per-chat-tab; cumulative and per-lever totals are GLOBAL (app-level).** `SavingsTracker` is `@Service(PROJECT)` and keys the running session band by `chatId` in a `chats` map (mirroring [CostTracker](cost-tracking.md)). But the persisted MTD/all-time `SavingsTotal` and the per-lever accumulator are stored in **application-level** `ClawDEASettings.state.savingsTotal["global"]` / `state.savingsByLever` — like `knowledgeUsd`, they read as one figure across every project and tab. Only the modeled levers accrue into `savingsByLever` (`accrueLevers` skips `measured` components) ([SavingsTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsTracker.kt), [LeverBandStore.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/LeverBandStore.kt), [SavingsTotal.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsTotal.kt)).
- **The section title is driven by the all-time expected net.** `SavingsSnapshot.isNetSaving` is `cumulative.allTime.expected >= 0.0` — the long-term verdict, not the session or MTD figure. It picks the panel title between "ClawDEA Estimated Savings" and "ClawDEA Estimated Cost". `isCollecting` (`turnCount < 2`) gates the "collecting… run a few turns" placeholder ([SavingsModels.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsModels.kt), [CostControlPanel.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/CostControlPanel.kt)).
- **Confidence is band width relative to magnitude.** `SavingsEstimator.confidence(band)` returns `ESTIMATE` when relative width (`|high - low| / |expected|`) `<= 0.5`, else `ROUGH`; a near-zero magnitude (`< 1e-6`) is always `ROUGH`. The panel labels the net figure "estimate" vs "rough estimate" from this ([SavingsEstimator.confidence](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsEstimator.kt)).
- **`SavingsTracker` publishes on the same topic as `CostTracker`.** `recordTurn`, `resetSession`, `seedFromResume`, and `resetCumulative` all fire `CostSnapshotListener.TOPIC.onCostChanged()`, so the Cost Control panel re-snapshots both cost and savings on one signal. Same re-entrancy contract as cost publishing — listeners must be quick (see [cost tracking](cost-tracking.md)) ([SavingsTracker.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/SavingsTracker.kt)).
- **Resume reconstructs the band from the transcript with the real re-ride multiplier.** `TranscriptSavingsReader.reconstructFile` splits the `.jsonl` into top-level turns (on `result` lines), builds a `TurnObservation` per turn with the accurate `remainingTurns`, and aggregates. It de-duplicates streamed duplicate lines by `message.id` (mirroring [TranscriptCostReader](cost-tracking.md)) — result lines and subagent lines each counted once. `SessionManager.resumeSession` calls it and hands the result to `SavingsTracker.seedFromResume(chatId, band, turns)`. Any exception swallows to a zero reconstruction — seeding is non-critical ([TranscriptSavingsReader.kt](../../../src/main/kotlin/com/adobe/clawdea/cost/TranscriptSavingsReader.kt), [SessionManager.kt](../../../src/main/kotlin/com/adobe/clawdea/chat/SessionManager.kt)).

## Resolution pipeline

1. **Accumulate per-turn signals.** As a turn streams, `EventStreamHandler` collects: index-tool hits (`turnIndexToolHits`, sized via `TurnObservationBuilder.estimateTokens` on each index tool's result), subagent observations (`turnSubagents` — each subagent's own cost, summary tokens, and the token size of files it read via `subagentReadTokens`), and the primer cache-read/creation token split.
2. **Build the observation.** On `CliEvent.Result`, after `CostTracker.recordTurn`, the handler reads `priorTurns = savingsTracker.snapshot(chatId).turnCount` as the re-ride proxy and calls `TurnObservationBuilder.build(model, remainingTurns = priorTurns, subagents, indexTools, primerCacheReadTokens, primerCacheCreationTokens, knowledgeUpkeepUsd)`.
3. **Estimate & accrue.** `SavingsTracker.recordTurn(chatId, obs)` runs `SavingsEstimator.components(obs)`, sums to the turn net, adds it to the chat's session band, folds the net into the global MTD/all-time `SavingsTotal` (`accrueCumulative`, calendar-month MTD rollover), accrues the modeled levers into `savingsByLever` (`accrueLevers`, skipping measured levers), stashes `lastComponents` (diagnostic), increments `turnCount`, then `publish()`es. The handler then clears its per-turn accumulators.
4. **Render.** `CostControlPanel.savingsCard(snapshot)` reads `SavingsTracker.snapshot(chatId)`, titles the section from `isNetSaving`, shows a per-lever breakdown (`rangeText` for the modeled bands) and a net figure with a confidence label; while `isCollecting` it shows the placeholder but **always renders the measured (real) cost** so a non-estimated cost is visible before the estimate converges.
5. **Resume.** `SessionManager.resumeSession` calls `TranscriptSavingsReader.reconstructFile` on the session transcript and seeds the chat via `SavingsTracker.seedFromResume` with the accurate reconstruction.

## Anti-patterns

- **Flooring the librarian lever at zero.** A one-shot question or an expensive-but-shallow subagent nets negative on purpose — that is an honest "ClawDEA cost more here" outcome. Clamping it would inflate the headline savings and break the sign convention.
- **Using the live `remainingTurns` as if it were accurate.** The live path only knows the *prior* turn count; it is a deliberate proxy floor. Only the transcript reseed path has the real future-turn count. Do not "fix" the live path to look ahead — the future genuinely isn't known at `Result` time.
- **Accruing measured levers into `savingsByLever`.** `accrueLevers` skips `measured` components on purpose: the per-lever accumulator holds only the modeled (estimated) levers. Primer/knowledge costs already flow through the cumulative `SavingsTotal` and (for knowledge) [cost tracking](cost-tracking.md)'s `knowledgeUsd`.
- **Treating the savings band as absolute spend.** It is a *delta vs. standard CC*, signed. Absolute per-turn dollars live in [`CostTracker`](cost-tracking.md); the two are separate accounting paths off the same events.
- **Pricing every transcript line during reconstruction.** Claude Code rewrites each streamed message to the `.jsonl` several times; `reconstruct`/`countTopLevelTurns` de-dup by `message.id`. Summing every line would over-count the reconstructed band, exactly as it would over-count resume cost.
